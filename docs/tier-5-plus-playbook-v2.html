Tier 5+ Playbook

Operação canônica para construir um site premium para um cliente, do briefing ao pós-launch. Cada fase tem prazo, definição de pronto e filtro do fundador. Atualizar a cada cliente novo.

Doutrina operacional

  1. Foco antes de adicionar. Toda inclusão exige uma exclusão equivalente. (Jobs)
  2. Artesanal nos primeiros 10 clientes. Você mesmo faz. Aprendizado > eficiência. (Graham — Do Things That Don't Scale)
  3. Bloco maker é sagrado. Construção real exige meio dia mínimo, sem reunião. (Graham — Maker's Schedule)
  4. Schlep é moat. A tarefa que ninguém quer fazer é o diferencial. (Graham — Schlep Blindness)
  5. Métrica única. Sites Tier 5+ entregues por mês — meta declarada. (Graham — Startup = Growth)
  6. Filtro do fundador em cada fase. Sem checkpoint, qualidade desliza. (Jobs — Soul of the Founder)

FASE 0Qualificação 90 min

Antes de aceitar, decidir se o cliente cabe em Tier 5+. Aceitar errado é mais caro que recusar. Esta fase é gratuita até "passa no filtro"; a partir daí, contrato.

Roteiro de briefing (22 perguntas — Graham: Do Things That Don't Scale)

Reunião 1:1 (Zoom focado, sem terceiros, 60 min). Você conduz. Pergunta-pivô no fim: "Qual é a frase de duas sentenças que um paciente seu fala quando finalmente decide entrar?" Anote palavra por palavra. Essa frase vira a headline do site.

  1. Qual é o nome legal e o nome de uso da marca?
  2. Qual problema do paciente vocês resolvem que ninguém mais resolve em <100km?
  3. Qual é o ticket médio do paciente alvo?
  4. Quantos pacientes ativos hoje?
  5. Qual a taxa de conversão do primeiro contato → primeira consulta?
  6. Quem são os 3 concorrentes que mais te incomodam — e por quê?
  7. Qual é a "frase de 2 sentenças"? (a pergunta-pivô)
  8. Quais são os pilares (3 a 5) — nomes que aparecem no menu?
  9. Quem são os profissionais? Foto, formação, especialidade.
  10. Existe blog hoje? Quem escreve? Frequência?
  11. Existem fotos profissionais da equipe? E do espaço?
  12. Existe mood board, identidade visual, manual de marca?
  13. Quais cores e tipografias estão definidas?
  14. Qual domínio? Já comprado? DNS em qual registrador?
  15. Existe GTM, Pixel, Analytics? IDs?
  16. Existe formulário de agendamento — para onde vai o lead?
  17. Existe LGPD/política de privacidade redigida por advogado?
  18. Quem aprova copy do lado do cliente? Uma pessoa ou comitê?
  19. Prazo desejado para go-live?
  20. Orçamento aprovado — número, não faixa.
  21. O que NÃO pode aparecer no site (compliance, ética, CFM)?
  22. Como vai medir sucesso 90 dias pós-launch?

Filtro do fundador — passa ou não passa

  1. Ticket médio do cliente está em ≥ R$ 35k? Se não, é projeto Tier 3, não Tier 5+.
  2. Tem aprovador único do lado do cliente, com decisão final? Se é comitê, recusar ou cobrar +50%.
  3. Domínio já está registrado e DNS é controlável? Se não, dia 1 é resolver isso.
  4. Cliente entrega copy ou eu produzo? Definir agora — muda preço em 2x.
  5. Cliente tem identidade visual mínima (logo, paleta, tipografia)? Se não, projeto começa por brand book separado.

Sinais de recusa — não aceitar mesmo com cheque na mesa

  • Cliente quer "site igual ao da clínica X" — copia, não premium.
  • Cliente quer área de membro, e-commerce, agendamento online complexo, integração com prontuário. Fora de escopo Tier 5+.
  • Cliente trabalha com seguradora popular como canal principal — desalinhamento de tier.
  • Comitê de 4+ aprovadores. Tempo de revisão vai matar margem.
  • Prazo < 21 dias corridos.

FASE 1Setup 30 min

Ambiente local pronto. Se passar de 30 min, há problema externo (Node, conflitos de merge, repositório errado) — diagnosticar antes de seguir.

# 1. Fork via interface GitHub. Clonar localmente:
git clone git@github.com:<sua-org>/<cliente-slug>.git
cd <cliente-slug>
# 2. Garantir Node ≥ 22.12.0 ANTES de npm install.
node --version  # se <22, usar nvm:
nvm install 22 && nvm use 22
# 3. Instalar e subir dev:
npm install
npm run dev      # http://localhost:4321

Definição de pronto — fase 1

  • npm run dev sobe sem erro.
  • http://localhost:4321/ renderiza homepage com placeholders TODO_*.
  • Repositório em branch setup/<cliente>, não main.

Filtro do fundador — fase 2

  1. A tagline é a "frase de 2 sentenças" do briefing, ou genérica?
  2. Cores e tipografias em tokens.css refletem o brand book do cliente, não os defaults?
  3. Os pilares têm nomes que um paciente entende, ou jargão clínico?
  4. O endereço está completo o bastante para o Google Maps gerar pin correto?
  5. Daqui a 5 anos, estes pilares ainda fazem sentido para a marca?

Definição de pronto — fase 2

  • Zero TODO_* em src/data/site.ts.
  • Tokens (cor, tipografia, espaço) em src/styles/tokens.css revisados.
  • Schema.org JSON-LD validado em Rich Results Test (rodando via build preview).

FASE 3Equipe — professionals.ts 2h / 6 pessoas

Cada profissional é um card. Foto ruim → marca degrada. Esta é fase de schlep: ninguém quer otimizar foto. Você faz.

{
  slug: 'dra-maria-silva',
  name: 'Dra. Maria Silva',
  role: 'Endocrinologista',
  unit: 'longevidade',          // slug do pilar
  photo: '/team/dra-maria.webp',
  bg: '#C4BBB8',
  description: 'Especialista em...',
  tags: ['Endocrinologia', 'Metabolismo'],
  education: { graduation: 'USP', postgrad: ['Residência HC'] },
  specialties: ['Diabetes', 'Tireoide'],
  mainComplaints: ['Cansaço', 'Ganho de peso'],
}

Padrão fotográfico — sem exceção

  • Fundo neutro (branco, cinza claro ou cor da marca). Sem fundo de paisagem, sem janela atrás.
  • Recorte de meio-busto. Olhar direto. Sorriso natural, não performado.
  • Resolução fonte: ≥ 2000px lado maior. Você reduz para entrega.
  • Entrega: nome.webp (480px, <100KB) + nome.avif (480px, <80KB).
  • Ferramenta: Squoosh ou sharp via npx.

Schlep alert — onde está o moat

Concorrentes deixam foto em JPG 800KB direto do iPhone. Você gasta 15 min por pessoa otimizando. Cliente não percebe diretamente, mas o Lighthouse percebe — e o Google rankeia. Aqui é onde Tier 5+ se separa de Tier 3.

Definição de pronto — fase 3

  • Toda foto carrega em <100ms na rede 4G simulada (Lighthouse).
  • Toda foto tem alt descritivo (não "imagem de Dra. Maria").
  • Card renderiza com bg correto da marca.
  • Campo unit bate com algum slug em PILLARS.

FASE 4Conteúdo — páginas .astro 4h

Substituir todos os TODO_* em src/pages/ por copy real. Esta é a fase mais sensível à voz da marca. Bloco maker — sem reunião nesse dia.

Ordem de ataque

  1. index.astro — homepage. Headline, sub, CTAs, depoimentos.
  2. Páginas de pilar — duplicar pilar-template.astro uma vez por pilar.
  3. sobre.astro — história curta. 3 parágrafos máximo.
  4. profissionais.astro — gerada do professionals.ts; só ajustar texto de abertura.
  5. contato.astro, agendamento.astro — formulários + texto de apoio.
# Duplicar template de pilar:
cp src/pages/pilar-template.astro src/pages/longevidade.astro
# Editar TODOs do novo arquivo + adicionar entry em PILLARS e NAV_LINKS.

Filtro do fundador — fase 4 (voz da marca)

  1. Eu, como paciente do tier do cliente, leria isso e marcaria a consulta?
  2. A homepage promete uma transformação concreta, ou descreve o que a clínica faz?
  3. Existe pelo menos uma frase no site que NENHUM concorrente local conseguiria copiar sem soar falso?
  4. O texto fala "você" para o paciente, ou "nós oferecemos"?
  5. Tem jargão clínico desnecessário ("abordagem multidisciplinar integrativa")? Cortar.

Definição de pronto — fase 4

  • Zero TODO_* em qualquer src/pages/*.astro.
  • Cada pilar tem sua página dedicada e aparece em NAV_LINKS.
  • Homepage testada em mobile real (não emulador) — headline cabe sem quebra estranha.

FASE 5Blog — seed inicial 2h ou cliente entrega

3 posts mínimos no go-live. Menos que isso, blog parece abandonado. Decisão Default Alive: se cliente não vai produzir conteúdo, não inclua blog no Tier — deixe oculto.

Configuração

Editar enum em src/content.config.ts com categorias reais (3 a 5):

category: z.enum(['Longevidade', 'Metabolismo', 'Hormonal'])

Frontmatter padrão

---
title: "Título claro de 6-9 palavras"
description: "Resumo de 1 frase, <160 chars, com keyword."
date: "2026-05-01"
author: "Dra. Maria Silva"
category: "Longevidade"
tags: ["sono", "envelhecimento"]
draft: false
---
Conteúdo em markdown. Headings H2/H3.

Decisão Default Alive — incluir blog?

Se o cliente não tem rotina editorial confirmada (autor + frequência + revisão), blog vai apodrecer em 6 meses e prejudica SEO. Opções:

  • Sim: 3 posts seed + agenda editorial assinada por cliente.
  • Não: remover rota /blog e link do nav. Mais honesto que blog morto.
  • Adicional Tier 5+ premium: 12 posts produzidos por copywriter (R$ 6-12k extra).

FASE 6Assets 3h

Fontes, favicon, OG image, manifest. Pequeno em volume, alto em impacto. Aqui o Lighthouse mata projetos descuidados.

Fontes self-hosted

  • Arquivos em public/fonts/: arboria-300.woff2, arboria-400.woff2, arboria-500.woff2, playfair-italic.woff2 (ou família escolhida no brand book).
  • Licença comprada em MyFonts ou Fonts.com — guardar PDF da licença na pasta do projeto.
  • Sem essas, BaseLayout.astro 404 nos preloads e CLS sobe.

Favicon, OG, manifest

  • public/favicon/favicon.svg — minimal, 1 cor, escalável.
  • public/og-default.jpg — 1200×630, <200KB, com logo + tagline.
  • public/manifest.json — atualizar name, short_name, theme_color.
  • public/robots.txt — apontar para sitemap real.

Definição de pronto — fase 6 (auditável)

  • Favicon visível na aba do iOS Safari (testar no iPhone real).
  • OG image renderiza no preview do WhatsApp e no opengraph.xyz.
  • Fontes carregam em <100ms (Lighthouse → Network).
  • Manifest validado em manifest-validator.
  • Lighthouse PWA score ≥ 90.

FASE 7QA Lighthouse 2h

Build local + thresholds. Sem isso, deploy é roleta. Bloco maker — não interromper.

npm run build       # gera dist/
npm run preview     # serve dist em :4321
# Lighthouse CLI:
npm install -g @lhci/cli@latest
lhci autorun        # usa lighthouserc.json do repo

Definição de pronto — fase 7 (thresholds Tier 5+)

  • Performance ≥ 0.90 (mobile e desktop).
  • Accessibility ≥ 0.90.
  • SEO ≥ 0.95.
  • Best Practices ≥ 0.95.
  • LCP ≤ 2500ms · CLS ≤ 0.1 · TBT ≤ 300ms.
  • Zero erros de console no preview.
  • Todas as páginas testadas (não só home).

Se falhar — onde olhar primeiro

  1. LCP alto → imagem hero não otimizada ou não preload.
  2. CLS alto → fonte sem font-display: swap ou imagem sem width/height.
  3. TBT alto → script de terceiro carregando síncrono. CSP pode estar permitindo isso.
  4. SEO < 0.95 → meta description faltando, alt em imagem, hreflang errado.

FASE 8Deploy 1h

Domínio + GitHub Pages + Cloudflare + HSTS. Cada passo tem ponto de falha — checar.

Sequência

  1. Em astro.config.mjs: substituir site: pela URL real (sem trailing slash).
  2. git push origin main — workflow deploy.yml dispara.
  3. GitHub → Settings → Pages → confirmar source = branch gh-pages.
  4. Domínio custom: adicionar CNAME em public/ com domínio nu.
  5. Cloudflare: SSL/TLS = Full (strict), Always Use HTTPS = On, HSTS max-age = 31536000.
  6. DNS: A record para IPs do GitHub Pages (185.199.108.153 …) ou CNAME para <org>.github.io.
  7. Aguardar propagação (até 24h, geralmente <30 min via Cloudflare).

Definição de pronto — fase 8

  • HTTPS verde no domínio final.
  • Redirect http:// → https:// automático.
  • Redirect www → apex (ou apex → www, decidido na fase 0).
  • Sitemap acessível em /sitemap-index.xml.
  • Lighthouse rodado contra produção (não localhost) — thresholds mantidos.

FASE 9Handoff 90 min

Cliente recebe não só o site — recebe o playbook de manutenção. Esta fase garante que o cliente não vai te ligar em 30 dias com pergunta básica.

Entregáveis no handoff

  • Vídeo Loom (15-20 min): tour pelo repositório, como editar copy, como adicionar post de blog, como publicar.
  • Documento PDF (3-5 páginas): domínio, registrador, DNS, GitHub, Cloudflare, GTM, Pixel — todos os logins/IDs.
  • Agenda de revisão: reunião de 30 min em D+30, D+90 — revisar métricas e ajustes.
  • Contrato de manutenção: opcional, R$ 1,5-3k/mês, inclui 2h/mês de updates.

FASE ∞Pós-launch 2h/mês

Manutenção contínua. Aqui é onde a maioria abandona — e por isso há moat (Schlep Blindness).

  • D+7: Submeter sitemap ao Search Console. Validar schema em Rich Results.
  • D+30: Revisar Core Web Vitals (Search Console). Testar formulários end-to-end. Confirmar GTM e Pixel disparando eventos corretos.
  • D+90: Lighthouse run contra produção. Backup do repo + Apps Script.
  • Mensal: 1 post de blog (ou cliente). Update de pacotes não-breaking. Verificar erros 404 em logs.
  • Trimestral: Revisão de copy obsoleta. Análise de queries do Search Console — ajustar headings.

Não fazemos — escopo fechado

Lista canônica do "fora de escopo Tier 5+". Comunicar na fase 0. Atualizar a cada cliente que tentou empurrar item novo.

  1. Área de membro / portal do paciente autenticado.
  2. E-commerce (gateway, carrinho, checkout).
  3. Agendamento online com integração de calendário do médico.
  4. Integração com sistema de prontuário eletrônico (Tasy, MV, Doctoralia).
  5. Aplicativo nativo iOS/Android.
  6. Sistema de cupom, programa de fidelidade.
  7. Multi-idioma além de pt-BR (custo separado).
  8. Integração com seguradora popular (Amil, Bradesco Saúde popular, etc.).
  9. Site para grupo de clínicas com mais de 3 unidades.
  10. Marketplace de profissionais (cliente vende espaço para outros médicos).
  11. Newsletter com sistema próprio (usar Resend, Substack, ConvertKit).
  12. Telemedicina embarcada (vídeo-consulta).

Perguntas-espelho antes de cada projeto

Filtrar antes de aceitar

  1. Este cliente cabe em qual coluna do Tier? Decidi antes de cotar?
  2. O que vai SAIR da minha agenda para este projeto entrar?
  3. Qual fase deste projeto é o schlep — onde construo moat?
  4. Estou cotando para mostrar atividade, ou para entregar transformação?
  5. Daqui a 12 meses, este site vai parecer prematuro ou correto?
  6. Se eu não pudesse aprovar pessoalmente, alguém da equipe defenderia esta entrega com o mesmo rigor?
  7. Ecooa-builder está default alive ou default dead se eu aceitar este projeto na margem cotada?
